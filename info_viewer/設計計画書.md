## 3. 完成イメージ
最終的な `info_viewer` は、YouTube 情報閲覧に特化した 3 ペイン UI とする。

- 左ペイン: チャンネル一覧
- 中ペイン: 動画一覧
- 右ペイン: Markdown 記事表示

データの考え方:

- Google Sheets の `チャンネル設定` を使い、viewer 対象チャンネルを決定
- Google Sheets の `動画リスト` を使い、各チャンネルに属する動画一覧を構成
- OneDrive 上の Markdown を既存 API 経由で取得して表示
- GitHub Variables に保存済みの `YT_SOURCE_*` を使い、記事と元動画の対応補助に使う

## 7. 第一弾の API 設計
### 7.1 新規 API の役割
新規 API は viewer 用の集約専用エンドポイントとする。

1. Google Sheets から対象チャンネルと動画一覧を読む

4. viewer がそのまま描画できる JSON に整形する

```json

      "videos": [

          "videoUrl": "https://...",
          "title": "動画タイトル",
          "publishedAt": "2026-04-08",
          "sheetStatus": "未完了",
          "hasArticle": true,
          "articleId": "onedrive-file-id",
          "articleTitle": "記事タイトル",
          "articleUpdatedAt": "2026-04-08T12:34:56Z",
          "youtubeUrl": "https://..."
        }
      ]
    }
  ]
}
```

## 10. リスクと対策
### 10.1 `public/index.html` の破壊リスク
最大リスクは、`C:\Users\HCY\OneDrive\開発\Blog_Vercel\public\index.html` の既存機能を巻き込むこと。

対策:

- 第一弾では変更箇所をリンク追加に限定
- 既存関数名の変更禁止
- 既存 UI レイアウト変更禁止
- 新機能は `info_viewer` 側へ閉じ込める

### 10.2 localStorage による新リンク未表示
既存利用者は `sb_customLinks` を保持している可能性がある。

対策:

- 初期リンク配列への追加だけで終わらせない
- 読み込み時に `info_viewer` リンクを補完する最小ロジックを実装する

### 10.3 Google Sheets 列名差異
本番シートの列名が設計と完全一致しない可能性がある。

対策:

- 実装時に存在チェック
- 欠損列を UI と API で明示
- 必須列と任意列を分ける

### 10.4 動画と記事の突合失敗
記事ファイル名だけで動画と厳密一致しない可能性がある。

対策:

- `YT_SOURCE_*` を使う
- URL 正規化する
- 一致しない場合も動画一覧表示を優先し、記事なし扱いで落とさない

### 10.5 API コスト
Apify / Gemini はコストや無料枠制限がある。

対策:

- 第一弾では自動取得を実装しない
- 第二弾で遅延処理モデルを導入する
