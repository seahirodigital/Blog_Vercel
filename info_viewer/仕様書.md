## 2. システム概要
`info_viewer` は、YouTube 動画一覧を Google Sheets から取得し、対象動画の文字起こしを Apify で取得し、Gemini で記事形式の Markdown に整形し、OneDrive に保存し、その結果を viewer で一覧・閲覧できる仕組みである。

役割は次の 3 層に分かれる。

- viewer 層: 一覧表示・詳細表示を行う
- API 層: manifest と記事本文を返す
- pipeline 層: Sheets 同期、Apify、Gemini、保存、manifest 更新を行う

## 7. API 仕様
### 7.1 `GET /api/info-viewer`
役割:

- 最新 `manifest.json` を返す

挙動:

- 候補保存先が複数ある場合でも、最新日時の manifest を優先する
- 単純な「最初に見つかったファイル」優先ではない

### 7.2 `GET /api/info-viewer?id=<articleId>`
役割:

- 指定記事の Markdown 本文を返す

挙動:

- frontmatter は除去して返す
- manifest 上の解決結果から対象記事を探す

### 7.3 `POST /api/trigger-info-viewer`
役割:

- GitHub Actions を起動するための API

主な入力:

- `mode`
- `max_items`
- `channel_name`
- `video_url`
- `rebuild_manifest_only`

注意:

- 現行運用では本線 workflow を起動するよう扱う
- legacy workflow を前提にした判断はしない

## 8. pipeline 実行モード仕様
### 8.1 `sync_only`
目的:

- Google Sheets と queue state を同期する

処理内容:

- Sheets 差分取得
- queue 同期
- `pipeline_state.json` 更新
- `manifest.json` 更新

行わない処理:

- Apify 文字起こし取得
- Gemini 整形
- 新規記事出力

### 8.2 `process_queue`
目的:

- queue に積まれた未処理動画を記事化する

処理内容:

1. `pipeline_state.json` から処理可能動画を抽出
2. Apify で文字起こし取得
3. Gemini で Markdown 整形
4. OneDrive に記事保存
5. Sheets の状態更新
6. `manifest.json` 更新

### 8.3 `full`
目的:

- 同期と処理を一続きで実行する

処理内容:

- `sync_only` 相当
- 続けて `process_queue` 相当

### 8.4 `rebuild_manifest_only`
目的:

- 既存成果物から `manifest.json` を再構築する

用途:

- viewer 表示不整合の修復
- 記事ファイルはあるが一覧が崩れた場合の再生成

## 10. Gemini 仕様
### 10.1 役割
Gemini は文字起こしを viewer 用の記事に整形する。  
単なる要約ではなく、Markdown として読みやすく再構成する役割を持つ。

### 10.2 モデル・呼び出し
現行では `gemini-2.5-flash` を基準とする。  
transport は `models.generate_content` を使用する。

### 10.3 実行制御
1 動画につき順番に候補キーを試す。  
キー間の連続呼び出しは一定の待機時間を置く。

### 10.4 retryable error の扱い
少なくとも次は retryable として扱う。

- `503 UNAVAILABLE`
- `429 RESOURCE_EXHAUSTED`
- 一部の transport エラー

挙動:

- 同一 run 内で次候補キーへフォールバックする
- それでもだめなら `deferred` または `failed` を状況に応じて記録する

### 10.5 quota 制限時の扱い
quota 系失敗時は、即座に「壊れた」とはみなさない。  
`recommendedWaitSeconds` と `nextRetryAt` を使って次回 run に送る。

## 11. queue / state 管理仕様
### 11.1 queue の考え方
`info_viewer` は一括即時処理ではなく、queue を介して段階的に記事化する。  
そのため、Sheets に新規動画が追加されても、常にその場で記事になるとは限らない。

### 11.2 `deferred`
`deferred` は失敗ではなく待機状態である。  
主な発生要因:

- Gemini quota
- 一時的 API 障害
- 再試行待ち

### 11.3 `failed`
`failed` は現 run での処理失敗を意味する。  
ただし viewer 上での見え方と、内部再試行対象かどうかは分けて考える。

### 11.4 `done`
以下まで完了した状態:

- 記事 Markdown 保存
- Sheets 状態更新
- manifest 反映
