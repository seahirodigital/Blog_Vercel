### 7.3 sync_only の仕様
`sync_only` は記事生成をせず、差分だけをキューへ反映する。

処理内容:

- Google Sheets の対象動画を読む
- OneDrive の既存記事一覧を読む
- 既存記事がある動画は `状況=完了` に同期する
- 新規動画を `pipeline_state.json` に登録する
- 既に存在しなくなった動画は `inactive` 化する
- `manifest.json` を更新する

### 7.4 process_queue の仕様
`process_queue` はキューから処理可能な動画だけを取り出して記事化する。

処理順:

1. `pipeline_state.json` から処理対象を選ぶ
2. Apify で文字起こし取得
3. Gemini で Markdown 整形
4. OneDrive に保存
5. Google Sheets の `状況` を `完了` に更新
6. `manifest.json` を更新

### 7.5 失敗時の挙動
失敗時はステージ別に情報を保持する。

保持対象:

- `Apify`
- `Gemini`
- `OneDrive保存`
- `状況更新`

保持場所:

- `manifest.json` の `failures`
- `manifest.json` の `processingLogs`
- `pipeline_state.json` の `lastError`
- `pipeline_state.json` の `lastStage`

### 7.6 quota 到達時の挙動
Gemini の quota 到達時は以下の仕様で動く。

- 現在処理中の動画を `deferred` 扱いにする
- 同 batch の残件も `deferred` にする
- `recommendedWaitSeconds` を見て次回再試行時刻を計算する
- 次回 run まで処理を打ち切る

### 12.2 Gemini キー分離と quota 時の挙動

2026年4月10日現在、Gemini キーの使い分けは「本編記事」と「info_viewer」で分離している。
特に把握しておくべき点は、ブログ本編では 4 キー構成、invest 系では 2 キー構成、
tech 系では 1 キー構成であり、`info_viewer` は本番記事フローの
`GEMINI_API_KEY` と `GEMINI_TOKEN_SUB3` を使用しないことである。

原稿フロー別の Gemini キー一覧は次の通り。

| カテゴリ | 原稿・フロー | 用途 | 使用キー数 | 使用キー | 切り替え順 | 補足 |
| --- | --- | --- | --- | --- | --- | --- |
| 本編 | 本編ブログ | ブログ本編の記事生成 | 4 | `GEMINI_API_KEY` / `GEMINI_TOKEN_sub` / `GEMINI_TOKEN_SUB2` / `GEMINI_TOKEN_SUB3` | `GEMINI_API_KEY -> GEMINI_TOKEN_sub -> GEMINI_TOKEN_SUB2 -> GEMINI_TOKEN_SUB3` | 実装は `C:\Users\HCY\OneDrive\開発\Blog_Vercel\scripts\pipeline\main.py` |
| Info Viewer | 投資ブログ | invest 系の viewer / 原稿整形 | 2 | `GEMINI_TOKEN_invest` / `GEMINI_TOKEN_INVESTSUB` | `GEMINI_TOKEN_invest -> GEMINI_TOKEN_INVESTSUB` | `info_viewer` の invest プール |
| Info Viewer | tech 系 | tech 系の viewer / 原稿整形 | 1 | `GEMINI_TOKEN_tech` | `GEMINI_TOKEN_tech` のみ | `info_viewer` の tech プール |

挙動の要点を一覧にすると次の通り。

| 項目 | 本編ブログ | `info_viewer` |
| --- | --- | --- |
| 主な実装場所 | `C:\Users\HCY\OneDrive\開発\Blog_Vercel\scripts\pipeline\main.py` | `C:\Users\HCY\OneDrive\開発\Blog_Vercel\scripts\info_viewer\runner.py` |
| キー枯渇時の基本動作 | 同一 run 内で次の候補キーへ即時切替 | 同一 run 内では本編キーへ逃がさず、対象記事を `deferred` にして次回 run へ回す |
| `429` / quota / rate limit 時 | exhausted key を以後その run では再利用しない | `recommendedWaitSeconds` と `nextRetryAt` を付与して再試行待ちにする |
| `SUB3` の利用 | 利用する | 利用しない |
| `GEMINI_API_KEY` の利用 | 利用する | 利用しない |

補足ルールは次の通り。

- 本番記事フローでは `GEMINI_TOKEN_SUB2` が `429` / quota / rate limit に到達した場合、待機せず `GEMINI_TOKEN_SUB3` へ即時切り替える。
- `info_viewer` 側では本番記事フローの `GEMINI_API_KEY` へ逃がさず、専用キーの範囲だけで処理する。
- `429` / quota 到達時は同一 run 内でその記事を `deferred` とし、`recommendedWaitSeconds` と `nextRetryAt` を付与して次回 run へ回す。

`workflow_dispatch` または API から `video_url` を指定した手動 run では、対象動画 1 本を優先選択できる。
この手動指定時は retry 待機中の記事でも対象にできるよう調整済みである。

Actions 画面の `Run workflow` では次の `mode` を使い分ける。

- `process_queue`: キュー済み記事の Gemini 整形を実行
- `sync_only`: スプレッドシートと OneDrive の同期、manifest 再生成のみ
- `full_pipeline`: 同期と整形をまとめて実行
- `rebuild_manifest_only`: manifest の再構築のみ

単一動画を試験する場合は `video_url` に YouTube URL を入れて `process_queue` を実行する。
累積記事をまとめて回す場合は `video_url` を空欄のまま `process_queue` を実行する。
