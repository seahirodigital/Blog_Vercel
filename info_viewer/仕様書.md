1. プロジェクト概要
既存のApify、Gemini API、Vercel環境を流用し、YouTubeの文字起こしデータを構造化・可視化するビューアーを構築する。既存のツール（エディター等）との相互リンクを確立し、情報集約プラットフォームとしての利便性を高める。

2. システムアーキテクチャ・外部連携
既存の資産をフル活用し、以下のAPI・サービスをシームレスに連携させます。

データ取得 (Apify): 既存のApifyを用いてYouTubeの動画情報および文字起こしテキストを取得。

AI処理 (Gemini API): 取得したテキストの整形、要約、マークダウン形式への構造化を実行。

データ管理 (Google Sheets / OneDrive):

Google Sheets: チャンネル一覧、取得状況（完了/未完了）の管理。


OneDrive API: 整形済みマークダウンファイルのストレージとして利用。

ホスティング (Vercel): ビューアー本体のデプロイおよびフロントエンド実行環境。

3. 機能要件
3.1 ワークフロー（GitHub Actions）
定期フェッチ: GitHub Actionsにより一定時間ごとに起動。

Apify実行: スプレッドシートの「未完了」URLをApifyへ渡し、文字起こしデータを取得。

Gemini整形: 取得データをGemini API経由で成形。無料枠を維持するため、失敗時はステータスを保持し、次回の実行タイミング（夜間等）でリトライするロジックを実装。

保存: 処理完了後、OneDriveへマークダウンファイルを保存し、スプレッドシートの「状況」列を「完了」に更新。

3.2 フロントエンド UI/UX（info_viewer）
既存のサイドパネル型ダッシュボードをベースに、エディター機能を排した閲覧特化型デザインへ刷新します。

ナビゲーション構造（2階層サイドパネル）:

第1階層（メインパネル）: 「YouTube」カテゴリの下に、スプレッドシートで管理されている「各チャンネル名」をリスト表示。

第2階層（サブパネル）: チャンネル名クリック時に展開。当該チャンネルに紐づく「動画タイトル」と「投稿日」をリスト表示。

コンテンツ表示エリア:

タイトルクリックにより、右側のメインエリアにマークダウン形式で整形された記事を表示。

レスポンシブ対応:

モバイル端末では面積が限られるため、サイドパネルをドロワー形式（引き出し式）にするなど、記事表示エリアを最大化する設計。

3.3 既存システムとの統合
相互リンクの実装:

既存の index.html（リンク集）に info_viewer への導線を追加。

info_viewer 内にも、元のツールやリンク集へ戻るためのナビゲーションを設置。

4. デザイン変更・削除事項
エディター機能の削除: 閲覧専用とするため、編集用UI、保存ボタン、テキストエリア入力機能は全て削除。

UI構成の整理:

[左サイドパネル：チャンネル一覧] ＞ [中サイドパネル：タイトル一覧] ＞ [右メイン画面：マークダウン表示]

3カラム構成（または2カラム＋オーバーレイ）により、シームレスなブラウジングを実現。

5. 運用上の工夫（APIコスト対策）
無料枠の使い回し:

APIエラーやレート制限が発生した際は、即座に停止せず、エラーログを記録して次のActions実行サイクルへタスクを回す「遅延処理モデル」を採用。

課金が発生しないよう、リクエスト間隔の調整と夜間バッチの活用を徹底する。
