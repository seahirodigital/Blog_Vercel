# Xpost_blog 開発計画

## 1. 目的
`C:\Users\HCY\OneDrive\開発\Blog_Vercel\Xpost_Blog\仕様書.md` を基準に、既存の本編記事作成 UI と `info_viewer` の queue 設計を流用しつつ、Discord に保存した X URL を元にブログ記事化する `Xpost_blog` を独立実装する。

## 2. 重要方針
- 本編を壊さないため、Xpost 用の API・workflow・Python を分離する。
- Xpost の取得は `SocialData API` を主軸とし、`Apify` は採用しない。
- queue 同期は `15分ごと` とし、1 run あたりの取得は最大 `1件` に抑える。
- thread 展開は行わず、記事化に必要な最小取得だけを行う。
- Gemini 整形は `C:\Users\HCY\OneDrive\開発\Blog_Vercel\info_viewer\仕様書.md` の `12.2 Gemini キー分離と quota 時の挙動` に合わせ、`GEMINI_TOKEN_tech` 専用運用とする。
- 検証を止めないため、`GEMINI_TOKEN_tech` が quota 到達した場合のみ `GEMINI_TOKEN_INVESTSUB` へ一段 fallback する。
- `GEMINI_TOKEN_tech` と `GEMINI_TOKEN_INVESTSUB` の両方が使えない場合に限り `deferred` 化し、次回 run に回す。

## 3. 流用元
- UI 骨格: `C:\Users\HCY\OneDrive\開発\Blog_Vercel\public\index.html`
- OneDrive CRUD: `C:\Users\HCY\OneDrive\開発\Blog_Vercel\api\articles.js`
- queue/manifest 設計: `C:\Users\HCY\OneDrive\開発\Blog_Vercel\scripts\info_viewer\runner.py`
- Gemini tech 運用方針: `C:\Users\HCY\OneDrive\開発\Blog_Vercel\info_viewer\仕様書.md`

## 4. 実装フェーズ
### フェーズ1
- `C:\Users\HCY\OneDrive\開発\Blog_Vercel\scripts\xpost_blog` を新設
- Discord から X URL を取得する queue 同期を作る
- SocialData から元投稿ソースを取得して OneDrive へ保存する
- `GEMINI_TOKEN_tech` でブログ記事化する
- manifest を出力する

### フェーズ2
- `C:\Users\HCY\OneDrive\開発\Blog_Vercel\public\xpost_blog\index.html` を用意する
- 左から「記事一覧」「元投稿ビューア」「エディタ」「プレビュー」の構成にする
- 手動実行ボタンから `C:\Users\HCY\OneDrive\開発\Blog_Vercel\.github\workflows\xpost-blog-queue.yml` を起動できるようにする

### フェーズ3
- UI の色味を `C:\Users\HCY\OneDrive\開発\Blog_Vercel\Xpost_Blog\Discord_connect\reference\design_guideline.md` に寄せる
- 元投稿ビューアの改善
- フォルダ管理や補助操作の本編同等化

### フェーズ4
- `SocialData API` の低頻度安定運用へ寄せる
- queue 同期を `15分ごと`、取得を `1 run 最大1件` に固定する
- thread 展開は行わず、記事化に必要な tweet / article 取得だけを行う
- コスト表、取得間隔、1日あたりの想定件数を `C:\Users\HCY\OneDrive\開発\Blog_Vercel\Xpost_Blog\techreference_Xpost.md` に残す
